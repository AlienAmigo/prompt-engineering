## 🧠 1. Цепочка рассуждений (Chain-of-Thought / CoT)

**Суть:** Заставляете модель "думать вслух" перед тем, как дать ответ. Это резко повышает качество сложных логических задач.

**Когда использовать:** Деплой, отладка сложных багов, оптимизация алгоритмов, архитектурные решения.

> **Комментарий:** В продакшн-задачах часто безопаснее и практичнее просить **краткий план, критерии выбора и проверяемые допущения**, а не подробное «мышление вслух». Например: «Сначала дай 5–7 пунктов плана и список предположений, затем решение».

**Пример:**

> **«Мне нужно оптимизировать загрузку изображений в FlatList. Не давай готовое решение сразу. Сначала распиши по шагам:**
>
> 1. Проанализируй возможные причины тормозов
> 2. Сравни подходы (react-native-fast-image, ресайз на бэкенде, пагинация)
> 3. Оцени плюсы/минусы каждого подхода для моего кейса
> 4. Только потом предложи решение с кодом»

**Почему работает:** Модель не "гадает" ответ, а строит логическую цепочку, что снижает риск галлюцинаций.

## 🔄 2. Few-Shot промптинг (Обучение на примерах)

**Суть:** Даете модели 2-3 примера того, как вы хотите получать ответ, перед основным вопросом.

**Когда использовать:** Генерация однотипного кода (тесты, компоненты, экраны), написание документации в корпоративном стиле.

**Пример:**

> **«Мне нужно, чтобы ты писал тесты в определенном стиле. Вот пример того, как я хочу:**
>
> ✅ ХОРОШИЙ ПРИМЕР:
>
> ```javascript
> describe('Auth Slice', () => {
>   it('should set loading true when login pending', () => {
>     const action = { type: 'auth/login/pending' };
>     const state = authReducer(initialState, action);
>     expect(state.loading).toBe(true);
>   });
> });
> ```
>
> ❌ ПЛОХОЙ ПРИМЕР (так не надо):
>
> ```javascript
> test('login works', () => {
>   // ... плохой тест
> });
> ```
>
> **Теперь напиши тесты для нашего cartSlice по такому же паттерну.**»

**Почему работает:** Модель копирует паттерн, а не генерирует "среднюю температуру по больнице".

<a id="multi-step"></a>

## 🎯 3. Метод "Золотого сечения" (Golden Prompt) / Итеративная декомпозиция (Multi-step prompting)

**Суть:** Разбиваете сложную задачу на несколько последовательных промптов, где каждый следующий опирается на результат предыдущего.

**Когда использовать:** Разработка целой фичи, создание архитектуры приложения, миграция кода.

**Пример сессии:**

> **Промпт #1:** "Act as architect. Проанализируй наш project_context.md и предложи 3 варианта архитектуры для фичи 'чат'."
>
> **Промпт #2:** "Выбираю вариант #2. Теперь создай структуру папок и файлов для этого варианта."
>
> **Промпт #3:** "Отлично. Теперь для этой структуры напиши Redux slice с типизацией."
>
> **Промпт #4:** "Теперь создай главный компонент экрана чата, используя созданный slice."

**Почему работает:** Модель не захлебывается в токенах, каждый шаг получается глубже и качественнее.

<a id="red-team"></a>

## 🔍 4. Промптинг "Антагонист" / "Красная команда"

**Суть:** Просите модель найти проблемы в своем же решении или покритиковать код.

**Когда использовать:** Code review, поиск уязвимостей, стресс-тестирование архитектуры.

**Пример:**

> **«Я написал вот этот компонент (вставляю код). Теперь:**
>
> 1. Act as a QA engineer — найди 5 потенциальных багов
> 2. Act as a security expert — найди уязвимости
> 3. Act as a performance auditor — найди проблемы с производительностью
> 4. После этого дай финальную версию с исправлениями»

**Или проще:**

> **«Ты senior разработчик. А теперь представь, что ты — мой архитектор, который терпеть не может плохой код. Покритикуй это решение как можно жестче. Что можно улучшить?»**

**Почему работает:** Заставляет модель включить режим "скептика" и искать проблемы, а не хвалить.

<a id="blooms-taxonomy"></a>

## 🏗️ 5. Метод "Таксономии Блума" (Bloom's Taxonomy)

**Суть:** Заставляете модель работать на разных уровнях мышления: запомнить, понять, применить, проанализировать, оценить, создать.

**Когда использовать:** Обучение джуниоров, изучение новых технологий, принятие сложных решений.

**Пример:**

> **«Разбери тему "Redux Middleware" по уровням:**
>
> 1. (Знание) Что такое middleware простыми словами?
> 2. (Понимание) Как работает цепочка middleware в Redux?
> 3. (Применение) Напиши простой логгер-мидлвар
> 4. (Анализ) Сравни redux-thunk и redux-saga — когда что выбирать?
> 5. (Оценка) В нашем проекте сейчас thunk. Стоит ли мигрировать на saga? Почему?
> 6. (Создание) Спроектируй кастомный middleware для аналитики в нашем проекте»

<a id="reverse-prompting"></a>

## 📝 6. "Обратный промптинг" (Reverse Prompting)

**Суть:** Просите модель задавать _вам_ вопросы, чтобы собрать информацию, а потом отвечает.

**Когда использовать:** Когда сами не до конца понимаете, что нужно, или задача очень сложная.

**Пример:**

> **«Act as a business analyst. Мы хотим добавить в приложение "Push-уведомления". Я не уверен, какие именно. Задай мне 10 вопросов, чтобы собрать требования. После каждого моего ответа уточняй, если нужно. Когда соберешь всю информацию — предложи техническое решение.»**

**Почему работает:** Модель выступает в роли "вдумчивого аналитика", а не "исполнителя с полуслова", что критически важно на старте проекта.

<a id="multimodal"></a>

## 🔗 7. Мультимодальный промптинг (с файлами)

**Суть:** Комбинируете разные типы данных в одном промпте (код + скриншот ошибки + JSON + описание).

**Когда использовать:** Отладка, ревью дизайна, работа с легаси кодом.

**Пример (для Deepseek, который понимает файлы):**

> **«Я приложил:**
>
> 1. `error_log.txt` — логи с реальной ошибкой
> 2. `screenshot.png` — скриншот того, как выглядит баг
> 3. `component.tsx` — код проблемного компонента
> 4. `package.json` — версии библиотек
>
> **Задача:** Найди причину бага и предложи фикс. Обрати внимание на версии библиотек — возможно, конфликт.»

<a id="output-formatting"></a>

## 🧩 8. Метод "Персонализации ответа" (Output Formatting)

**Суть:** Жестко задаете формат, в котором хотите получить ответ.

**Когда использовать:** Интеграция с другими инструментами, создание документации, генерация конфигов.

**Пример:**

> **«Нужно создать Dockerfile для нашего React Native проекта (не Expo).**
>
> **Формат ответа:**
>
> 1. Сначала объясни стратегию (multistage? alpine?) — 2-3 предложения
> 2. Потом сам Dockerfile с комментариями на русском
> 3. Потом команды для сборки и запуска
> 4. В конце раздел "Возможные проблемы и их решения"
>
> Используй маркдаун для форматирования. Код оформи в отдельных блоках с подсветкой синтаксиса.»

<a id="socratic"></a>

## 🤔 9. Сократический метод (Socratic Prompting)

**Суть:** Модель не дает ответ, а задает наводящие вопросы, чтобы вы сами пришли к решению.

**Когда использовать:** Обучение, рефакторинг, когда хотите разобраться в проблеме, а не получить готовое решение.

**Пример:**

> **«Я пытаюсь реализовать infinite scroll в FlatList, но пагинация работает криво. Не давай мне готовый код. Задавай вопросы, которые помогут мне самому понять, в чем ошибка. Веди меня как ментор.»**

---

<a id="multi-perspective"></a>

## 📊 10. Метод "Множественных перспектив" (Multi-Perspective)

**Суть:** Просите модель оценить проблему с позиций разных специалистов.

**Когда использовать:** Принятие архитектурных решений, планирование спринта, оценка рисков.

**Пример:**

> **«Оцени идею переписать наше приложение с React Native на Kotlin Multiplatform.**
>
> - Как разработчик: оцени сложность, время, подводные камни
> - Как CTO: оцени бюджет, найм специалистов, долгосрочную поддержку
> - Как Product Manager: оцени влияние на скорость вывода фич
> - Как пользователь: заметит ли он разницу?
>
> В конце дай сводную рекомендацию: стоит ли овчинка выделки?»

<a id="anti-patterns"></a>

## ⛔ Анти-паттерны (чего избегать)

- **Слишком размытая цель**: «сделай красиво/быстро/лучше» без критериев успеха (метрики, ограничения, “готово когда…”).
- **Нет контекста**: версия стека, платформа, ограничения по деплою, архитектурные решения, пример кода/структуры проекта.
- **Отсутствуют входные/выходные данные**: нет примеров payload’ов, логов, ошибок, ожидаемого результата, edge cases.
- **Смешивание ролей и требований**: «будь архитектором» и одновременно «не предлагай варианты, сразу код», плюс конфликтующие ограничения.
- **Запрос “лучшее решение” без критериев**: не указаны стоимость внедрения, риски, сроки, требования по поддержке/команде.
- **Нет формата ответа**: из-за этого модель часто выдаёт “простыню” вместо структуры (план → решение → риски → тест-план).
- **Не просите проверку**: не запрашиваете допущения, риски и мини тест-план — повышается шанс галлюцинаций и “непроверяемых советов”.

## 🚀 Бонус: Комбо-промпт для реальной задачи

Соединяем несколько техник в одной задаче:

> **«Act as a senior React Native architect.**
>
> **Контекст:** проект на RN 0.72 + Redux Toolkit (файлы прикрепил).
>
> **Задача:** Нужно добавить офлайн-режим для корзины.
>
> **Техника Chain-of-Thought:** Сначала распиши возможные стратегии (AsyncStorage, SQLite, redux-persist). Сравни их плюсы/минусы для нашего проекта. Выбери лучшую и обоснуй.
>
> **Техника "Антагонист":** После того как предложишь решение, найди в нем 3 потенциальных проблемы.
>
> **Формат ответа:**
>
> - Используй маркдаун
> - Код на TypeScript
> - Комментарии в коде на английском
> - Объяснения на русском
>
> **Few-Shot пример (как я люблю):** Вот пример нашего существующего `authSlice`, делай похожий стиль.
> [пример кода authSlice]
>
> Готов? Начинаем с анализа стратегий.»

| ------------------------------------------------------------- | ------------------------------- | ------------------------------------------------------------------ |
| [**Ролевой промптинг**](#intro)                               | Всегда                          | Задает тон и экспертизу                                            |
